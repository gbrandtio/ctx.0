# ctx.0 UI/UX guidelines

These are the design standards for the Flutter app ctx.0 generates. They bind the
templates under `templates/mobile`: the base app, the navigation shells, and every
feature overlay. A generated workspace is a starting point someone else will build on,
so the UI it ships with has to look like one app rather than a pile of features, and it
has to keep working after the seed colour, the locale, or the shell is changed.

## 1. Core principles

- **One theme, no local styling.** Screens read colours and text styles from
  `Theme.of(context)`. A feature overlay never defines its own palette.
- **Stock Material 3.** Prefer the framework widget over a hand-built equivalent.
  Generated code is meant to be read and edited by the person who ran the CLI.
- **Overlay independence.** Features are copied on top of the base app in any
  combination, so no screen may depend on another feature's widgets or styles.
- **Localised and accessible.** Every user-visible string comes from `AppL10n`, and
  layouts survive long translations, large text scales, and both brightness modes.

## 2. Theme

The app's single theme is generated into `app/lib/app/theme.dart` from the colour scheme
and font chosen at create time, and `app.dart` binds both brightnesses to it:

```dart
theme: AppTheme.light(),
darkTheme: AppTheme.dark(),
```

`AppTheme` derives everything from one `seed` colour through `ColorScheme.fromSeed`, in
light and dark alike. Changing that seed is the supported way to rebrand a generated app,
which only works while templates take their colours from the scheme. The file is
regenerated by re-running the scaffolder, so edits to it are lost; the seed is the value
to change by hand.

- **Primary actions**: `colorScheme.primary`, applied by `FilledButton` and
  `FloatingActionButton` without configuration.
- **Errors**: `Theme.of(context).colorScheme.error`. Never `Colors.red`.
- **Surfaces**: leave `Scaffold`, `Card`, and `AppBar` at their defaults.
- **Typography**: `theme.textTheme` roles, `titleMedium` and `bodyMedium` for card
  content. The platform font is the default and adds no dependency. When a font was
  chosen, `AppTheme` merges that family's text theme over the Material 3 one through the
  `google_fonts` package, which is the only thing that puts the package in `pubspec.yaml`.

Both choices are optional, and both are recorded in the workspace manifest.

### Fonts and languages

Not every font covers every language the scaffolder offers (§7). Of the five, only Greek
needs glyphs beyond Latin, and several of the offered families ship none: Lato, Poppins,
Montserrat and Merriweather are Latin-only, while Inter, Roboto, Open Sans, Source Sans 3,
IBM Plex Sans, Noto Sans and Roboto Slab cover Greek as well. A shortfall is not fatal,
since Flutter falls back per glyph and the app still runs, but the uncovered language
renders in the platform font and stops matching the rest of the app. The catalog in
`packages/core/src/theme.ts` declares each family's coverage and the pickers say what a
choice would cost before it is made. Adding a language to the offered set means re-checking
that catalog against the Google Fonts metadata.

Feature overlays that need app-wide widgets attach at the anchors in `app.dart`:
`app-material` for `MaterialApp` configuration, `app-overlay` for anything drawn above
every route such as the consent banner, `home-wrap` for gates such as the auth gate.

## 3. Navigation shells

The root screen is `CtxShell`, generated from one of the shells in
`templates/mobile/shells`: `bottom_nav`, `nav_rail`, `drawer`, `home_list`. Feature
pages are inserted into the shell's `ctx:gen:pages` and `ctx:gen:destinations` anchors,
which means a page must be a self-contained widget that renders correctly in any of the
four shells. A page provides its own `Scaffold` and `AppBar`; it never assumes a
particular navigation chrome around it.

## 4. Screens

### Structure

Every feature page is a `Scaffold` with an `AppBar` whose title is a localised string.
State comes from a Cubit through `BlocBuilder` or `BlocConsumer`, and each page renders
the four states explicitly:

- **Loading with no data**: `Center(child: CircularProgressIndicator())`.
- **Failure with no data**: centred `Text` in `colorScheme.error`.
- **Empty**: centred localised sentence saying what the screen will hold.
- **Content**: the list or form.

An action already in flight disables its own control and shows progress in place, as
`ProfilePage` does when saving and `MediaPage` does when uploading. A failure that
arrives while data is already on screen goes to a `SnackBar` through `BlocConsumer`
rather than replacing the content.

### Lists

`ListView` or `ListView.builder` of `ListTile` rows: `leading` icon, `title`, optional
`subtitle`, `trailing` for the row's single action or its selected check. Lists backed
by a repository wrap in a `RefreshIndicator` bound to the Cubit's load method. Cards are
for content that sits above the app rather than in a list, the consent sheet being the
example; a card in a list is redundant with the row.

### Forms

`TextField` with `InputDecoration(labelText:)` and no border override, so the theme
supplies the shape. Set `keyboardType` and `textInputAction` to match the field.
Controllers are created in `State` and disposed. Long forms are a `ListView` so the
keyboard cannot overflow them. The submit control is a `FilledButton`; the secondary
choice next to it is a `TextButton`; a destructive choice is an `OutlinedButton` in
`colorScheme.error`, as on the privacy page.

## 5. Layout and spacing

- **Screen padding**: `EdgeInsets.all(16)`.
- **Vertical rhythm**: `SizedBox` in multiples of 8, using 8 between related fields, 16
  between groups, 24 before a submit button.
- **Full-width content**: `Column` with `crossAxisAlignment: CrossAxisAlignment.stretch`
  instead of sizing buttons by hand.
- **Overlays**: anything drawn over a route wraps in `SafeArea`, since it is not inside
  a `Scaffold` body.

## 6. Icons

Built-in Material icons only, outlined variants for row actions
(`Icons.delete_outline`). Templates ship no icon assets and no SVG dependency, which
keeps a generated app buildable without fetching anything beyond the packages in
`pubspec.yaml`.

## 7. Localisation

`AppL10n.of(context)` is read once at the top of `build`, conventionally into `l`. Every
label, title, hint, empty state, and error format goes through it, with the strings
declared in the feature's own `l10n/*.arb` files for each supported language. Errors use
the shared `commonError` template. The language picker names each language in that
language so it stays legible to someone who cannot read the current one.
