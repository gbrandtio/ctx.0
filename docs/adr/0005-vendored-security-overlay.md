# ADR-0005: The security plane is vendored source, always on

**Status**: Accepted — recorded retroactively, 2026-07-21.

## Context

ctx.0's premise is that the app and API are secure *by default* — encrypted at the
application layer, request-signed, device-enrolled. There were two decisions to make about
how that plane reaches the user.

**Library or source?** Shipping it as a package (a pub package plus a NuGet package) is the
conventional answer: one place to fix a bug, versioned upgrades. But it makes the most
security-critical code in the product the least visible part of the user's repository, and
it makes ctx.0 a permanent runtime dependency of every workspace it generates. A user
auditing "how is my traffic encrypted" would land in a package cache.

**Optional or mandatory?** Making it a catalog feature is symmetric with everything else.
But a security plane that can be switched off is a security plane that will be switched off
— and the base trees would then need to work both with and without it, doubling the
composition surface at exactly the point where mistakes are most expensive.

There is a third element: the encodings the plane uses (P-256 scalar and point formats,
base64) are part of the wire protocol. If a frontend generated the server keys itself, two
frontends could disagree, producing keys the API rejects.

## Decision

**The security plane is vendored source, applied unconditionally.**

- `templates/security/mobile` and `templates/security/api` are layers applied immediately
  after the base trees and before any feature. They carry `"core": true` and are **not** in
  the toggleable catalog: `loadCatalog` only scans `features/`, so there is no id to
  disable.
- The code lands in the workspace as ordinary source the user owns, reads, and can modify:
  `app/lib/security/` and `api/src/**/Security/`.
- They contribute `env` and `userSteps` like any feature, which is how a user learns which
  variables the API needs.
- **Server secrets are generated by the engine** (`packages/core/src/secrets.ts`), never by
  a frontend, because the encodings are protocol. `ctx0 keygen` only prints what the engine
  returns.
- Both overlays implement one specification — `protocol/wire-protocol.md` — and both assert
  against the same golden vectors.

## Consequences

- Every generated workspace has ALE, request signing, device enrollment, JWT with refresh
  tokens, PBKDF2 hashing, envelope encryption with blind indexes, row-level security and a
  RASP gate. There is no configuration in which it does not.
- The user can audit and adapt the security code without leaving their repository, and their
  app has no ctx.0 runtime dependency.
- The two implementations cannot silently diverge: the golden vectors are shipped into the
  workspace at `.ctx/vectors.json` and both test suites reproduce every intermediate value.
- **Cost**: no upgrade path. A fix in the security overlay does not reach workspaces already
  generated; their owners must port it. This is the price of ownership, and it is the
  sharpest cost of this decision.
- **Cost**: the plane is duplicated in two languages, so every protocol change is two
  implementations plus regenerated vectors.
- **Cost**: a user who genuinely does not want it must delete code rather than omit a flag.
